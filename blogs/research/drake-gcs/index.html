drake-gcs

A record for configure robotic simulator drake and GCS motion planning.

Drake is a newly developped robotic simulator by Toyota and MIT. The motivation to use it comes from Tobia Marcucci's paper "Motion planning around obstacles with convex optimization", which eaxctly used drake. They aslo implemented a quite mature reproduction in drake's libirary. However, drake is not well documented considering tutorials and examples and it's still a quite samll community, despite some quesitons in their github and stackoverflow.

The course materials from Prof. Tedrake himself at MIT is the best way to learn drake. We will mainly use the course Robotic Manipulation in this record. And also combine Prof. Wei Zhang's course on Advanced Control for Robotics at SUSTech, which is full of solid math and programming. The focus of this record is configure custom robots and GCS motion planning in drake.

Custom Robot and environment configure

The drake naturally supports sdf file and urdf file for robot configuring so it's easy to import robot arms. But I'm going to conduct a problem about continuum robot and motion planning. Some have also tried urdf description for continuum robot, it's quite similar with what people do for other simulators such as coppeliaSim or ROS. But I have one concern about calculation cost for such amout of nodes in a continuum robot. The FK is just the same as what the modeling of continuum robot, which is unavoidable. But the collision check may take more time. Hopefully, the GCS planning make a pre-planning decomposition of configuration space. So the pre-processing maybe long, but the acutal planning may not suffer from this.

Configuration space convex decomposition (IRIS)

My project originally focus on neuronavigation avoiding major functioning areas and cerebrobasculars. The vascular environemnt is highly complex, narrow annd non-convex. These natures inspired the use of continuum robot and GCS motion planning rather thant tranditional RRT or PRM or A*.

Machine translation is the problem of translating sentences from some source language to a target language. Neural machine translation (NMT) directly models the mapping of a source language to a target language without any need for training or tuning any component of the system separately. This has led to a rapid progress in NMT and its successful adoption in many large-scale settings.
Figure 2: Illustration of the machine translation problem.
NMT systems typically consist of two main components: the encoder and the decoder. The encoder takes as input the source language sentence and generates some latent representation for it (e.g., a fixed-size vector). The decoder then takes that latent representation as input and generates a sentence in the target language. The sentence generation is often done in an autoregressive manner (i.e., words are generated one-by-one in a left-to-right manner).
